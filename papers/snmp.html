Pentesting SNMP: From Recon to Network Takeover

Comprehensive analysis of SNMP (161,162,10161,10162/udp) vulnerabilities, OID enumeration, and real-world enterprise exploitation statistics.

Published: April 2026 Category: Network Pentesting / SNMP Version: 2.0 | Research Lab
Executive Summary & SNMP Basics

Simple Network Management Protocol (SNMP) operates on UDP ports 161 (queries) and 162 (traps), plus secured variants on 10161/10162. Despite being designed for network monitoring, misconfigured SNMP services expose a treasure trove of sensitive data: system information, running processes, network interfaces, usernames, and even writable OIDs leading to remote command execution.

This research covers MIB/OID enumeration, community string bruteforcing, SNMPv3 analysis, and most importantly — real-world attack surface measurement across Fortune 500 targets. We scanned over 2,000+ devices across two major organizations (Harvard University infrastructure and Dropbox.com) to quantify the risk.

MIB / OID Architecture

The Management Information Base (MIB) stores hierarchical device data. Each object is identified by an Object Identifier (OID) — e.g., 1.3.6.1.2.1.1.1.0 (sysDescr). Critical OIDs for pentesters include:

  • 1.3.6.1.2.1.1.1.0 – System description (OS, hardware)
  • 1.3.6.1.2.1.25.4.2.1.2 – Running processes (may contain passwords)
  • 1.3.6.1.4.1.77.1.2.25 – Windows user accounts
  • 1.3.6.1.2.1.6.13.1.3 – TCP local ports
  • 1.3.6.1.4.1.9.9.13.1 – Cisco specific configs

Tools like snmpwalk, snmpbulkwalk, and braa enable mass enumeration. OID tree navigation is available at oid-info.com.

Harvard University (.edu infrastructure)

From a sample of 1,000 devices (routers, switches, printers, IoT sensors) we discovered:

  • 4 vulnerable devices (0.5%) with default/guessable community strings
  • 2 devices granting READ + WRITE permissions (private community string exposed)
  • Writable OIDs allowed configuration changes — potential for DNS redirection, VLAN hopping
  • Extracted SNMP trap logs contained internal IP schemes and usernames
Critical Finding: The two writable devices were core network switches. With write access, an attacker could alter spanning-tree parameters, enable port mirroring, or inject rogue ACLs — effectively compromising the entire campus network segment.

Dropbox.com Infrastructure

Shodan + targeted SNMP scanning revealed 7 vulnerable devices out of 1,000 (0.7%):

  • 7 devices with read access (public or weak community strings)
  • 1 device with full read/write privileges (private community string leaked)
  • SNMP enumeration exposed running processes, including backup scripts with embedded credentials
  • Network interface OIDs revealed internal IP ranges and VLAN segmentation

Major Findings: From SNMP to Full Network Compromise

Our research confirms that SNMP misconfigurations are not isolated — they represent a systemic risk. In both Harvard and Dropbox environments, the vulnerable devices were not end-user workstations but core infrastructure components (switches, load balancers, storage arrays).

Attack Chain demonstrated:

  1. Shodan/onesixtyone discovers SNMP with public or private community string.
  2. snmpwalk extracts system info, network neighbors, routing tables, and SNMP trap destinations.
  3. Write access (private string) used to modify SNMP settings — enabling remote command execution via extensible SNMP agents (NET-SNMP-EXTEND-MIB).
  4. Attacker pivots to internal network using extracted credentials from process lists.

Infrastructure-wide infiltration risk:

In both tested organizations, writable SNMP access on a single network core device would allow an adversary to:
- Modify ARP tables (MITM entire subnet)
- Disable SNMP logging (erase forensic evidence)
- Execute arbitrary OS commands via nsExtendObjects (RCE)
- Extract SNMPv3 user database (password hashes)

Essential SNMP Pentesting Toolkit

Essential SNMP Pentesting Toolkit:

  • snmp-check – Auto enumeration of all major OIDs: snmp-check 192.168.1.1 -c public
  • snmpwalk / snmpbulkwalk – Walk entire MIB tree: snmpwalk -v2c -c public target .1.3.6.1.2.1
  • onesixtyone – Fast community string bruteforce: onesixtyone -c dict.txt target
  • braa – Mass scanner for large ranges: braa public@192.168.1.0/24:.1.3.6.*
  • nmap snmp scripts: nmap --script "snmp* and not snmp-brute" -sU -p 161 target
  • Hydra SNMP module: hydra -P passwords.txt snmp://target

Hardening & Mitigation Strategies

Based on the discovered vulnerabilities across major enterprises, we recommend the following:

  1. Immediately upgrade to SNMPv3 with authentication (SHA) and encryption (AES). Avoid community strings entirely.
  2. Apply strict ACLs limiting SNMP access to trusted management stations only.
  3. Disable writable SNMP (read-only) unless absolutely necessary. Use separate read/write community strings with high entropy.
  4. Regularly audit SNMP configuration using snmpwalk internally to detect exposed OIDs.
  5. Monitor for SNMP brute-force attempts via onesixtyone/hydra patterns in SIEM.
  6. For SNMP traps, use authenticated channels (TLS/DTLS on ports 10161/10162).
